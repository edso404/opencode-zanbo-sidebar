# Contributing to @oh-my-sidebar

Thanks for your interest in contributing! This document covers the workflow, conventions, and how to get started.

## Quick Start

```bash
git clone https://github.com/edso404/oh-my-sidebar.git
cd oh-my-sidebar

corepack enable
corepack prepare pnpm@11.9.0 --activate
pnpm install
pnpm build
```

## Development Workflow

1. **Fork and clone** the repository.
2. **Create a feature branch** from `main`:

   ```bash
   git checkout -b feat/my-feature
   ```

3. **Make your changes** — see conventions below.
4. **Run quality checks** before committing:

   ```bash
   pnpm check    # lint + typecheck
   pnpm test     # run tests
   ```

5. **Create a changeset** if your changes affect published packages:

   ```bash
   pnpm changeset
   ```

   Select the affected packages and bump type (patch/minor/major). This generates a markdown file in `.changeset/` that will be consumed during release.

6. **Commit** using conventional commit style (see below).
7. **Push and open a PR** against `main`.

## Commit Style

We follow [Conventional Commits](https://www.conventionalcommits.org/):

```
<type>: <short description>

[optional body]
```

Common types: `feat`, `fix`, `chore`, `docs`, `refactor`, `test`, `ci`.

Examples:

```
feat(context-progress): add cache hit ratio to progress bar
fix(session-tokens): handle missing modelID gracefully
docs: update README with new plugin instructions
```

## Pull Request Guidelines

- PRs should target **one concern** — a single feature, fix, or refactor.
- Keep the scope small. Large PRs are harder to review.
- Ensure CI passes (lint → typecheck → test → build).
- Add tests for new functionality when applicable.
- Update README or package documentation if the user-facing interface changes.
- If your PR introduces a visual change to a TUI widget, include a screenshot or ASCII capture if possible.

## Plugin Development

Each plugin is a standalone package under `packages/`.

### Structure

```
packages/<name>/
├── src/
│   └── index.tsx       # Plugin entry (default export: TuiPluginModule)
├── dist/               # Build output (gitignored, generated by pnpm build)
├── package.json        # Scoped to @oh-my-sidebar/opencode-<name>
├── tsup.config.ts      # Build config (ESM, JSX automatic)
├── tsconfig.json       # Extends tsconfig.base.json
└── README.md           # Package-specific documentation
```

### Conventions

- **Plugin ID:** `oh-my-sidebar.<name>` (kebab-case).
- **Export entry:** `./tui` → `dist/tui.tsx` in `package.json#exports`.
- **Slot:** Register under `sidebar_content` in the TUI plugin API.
- **Order:** Lower `order` numbers render first (context-progress: 100, session-tokens: 120).
- **Dependencies:** Use the pnpm workspace catalog for shared deps. Add new ones to `pnpm-workspace.yaml#catalog`.
- **Build:** The `build` script copies `src/index.tsx` to `dist/tui.tsx`. tsup config is available for type declaration generation.
- **No bundling**: Plugins are distributed as source TSX, not bundled. The OpenCode TUI compiles them at runtime.

### Adding a New Plugin

1. Copy `packages/context-progress` as a template.
2. Update `package.json`: `name`, `description`, `keywords`, `repository.directory`.
3. Implement your plugin in `src/index.tsx` with a unique plugin `id`.
4. Update the root `README.md` package table.
5. Register any new catalog dependencies in `pnpm-workspace.yaml`.
6. Create a changeset.

## Code Quality

- **Linting:** Biome (preset: recommended). Run `pnpm lint`.
- **Formatting:** Biome (2-space indent, 100 char width). Run `pnpm format`.
- **TypeScript:** Strict mode. Run `pnpm typecheck`.
- **Tests:** Vitest per package. Run `pnpm test`.

## Release Process

Releases are automated via GitHub Actions + Changesets:

1. A contributor merges a PR with pending changeset files.
2. The Release workflow creates a Version PR that bumps versions and updates changelogs.
3. Merging the Version PR publishes the updated packages to npm (OIDC trusted publishing).

See [ARCHITECTURE.md](./docs/ARCHITECTURE.md) for CI/CD pipeline details.
